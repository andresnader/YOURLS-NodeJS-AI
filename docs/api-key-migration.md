# API Key Hashing Migration

API keys are now stored as SHA-256 hashes instead of plaintext. New keys
generated by `POST /api/keys` are returned to the client **exactly once**
in the response and never recoverable from the DB.

## Schema changes

`ApiKey` model gained:

- `keyHash String? @unique` — SHA-256 hex of the raw key. Indexable for O(log n) lookup.
- `keyPrefix String?` — first 11 chars (e.g. `yn_a1b2c3d4`) for UI display.
- `expiresAt DateTime?` — optional expiration.

The legacy `key` column is now nullable. It will be removed in a follow-up
migration once all rows have been backfilled.

## Deployment steps

1. **Push schema** (creates the new columns, leaves existing rows intact):

   ```bash
   npx prisma db push
   ```

2. **Backfill existing keys** (hash plaintext into `keyHash`, fill `keyPrefix`):

   ```bash
   npx tsx scripts/backfill-api-key-hash.ts
   ```

   Idempotent — safe to re-run.

3. **Verify** that existing integrations still work (WordPress plugin, any
   CI/CD calling `/api/shorten` with `x-api-key`). The lookup path tries
   hash first, then falls back to the legacy plaintext column.

4. **Null out plaintext** once verified:

   ```bash
   npx tsx scripts/drop-plaintext-api-keys.ts
   ```

5. **(Optional, later)** Drop the `key` column entirely in a follow-up
   schema change.

## Operational notes

- Setting `SESSION_SECRET` is required in production for cookie auth to
  work. Generate one with `openssl rand -hex 32` and set it in Railway.
- API keys are revoked by setting `isActive=false` (already supported) or
  by deleting the row. Rotation is a delete + create flow.
- The master key fallback via `API_KEY` env var still works for emergency
  access. Remove it once normal keys are migrated.
